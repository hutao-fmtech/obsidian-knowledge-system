## 方法概述

收件箱清空是个人知识管理系统中最关键的日常习惯之一。收件箱（Inbox）是临时捕获区，用于快速记录想法、保存资料、记录任务，但**不是存储的地方**。定期清空收件箱，将内容分类归档到正确位置，是保持系统健康运转的核心动作。

本方法基于 GTD（Getting Things Done）理念，结合 PARA 分类方法，提供快速决策框架，帮助你在每日 15 分钟内清空收件箱，避免信息堆积和决策疲劳。

核心理念：**收件箱是缓冲区而非存储区，每日归零是系统健康的标志。快速决策优于完美分类。**

关键原则：**固定时间清理、快速决策不犹豫、批量处理同类项、2分钟立即行动、零收件箱目标。**

## 适用场景

- 每日晚间例行清理（最常见）
- 收件箱堆积超过 20 条时的紧急清理
- 每周回顾时的深度整理
- 快速笔记积累过多
- 待处理资料长期未归档
- 灵感闪念需要提炼
- 临时任务需要规划
- 会议记录需要拆解

## 具体步骤

### 步骤 1：设定固定的清理时间

**原则**：清空收件箱应该是**习惯**，而非随意行为。

**推荐时间段**：

**方案 A：每日晚间清理**（推荐）
- **时间**：21:00-21:15（15 分钟）
- **频率**：每天
- **优势**：
  - 一天结束时大脑较放松，适合整理
  - 当天的内容记忆清晰，决策快速
  - 为第二天创造清爽的开始
- **适合人群**：有规律作息，晚上有固定时间

**方案 B：每日午间清理**
- **时间**：12:30-12:45（15 分钟）
- **频率**：每天
- **优势**：
  - 午休时间的短暂休息
  - 上午捕获的内容及时处理
  - 下午工作前心理清空
- **适合人群**：午休时间充裕，喜欢分段处理

**方案 C：每日两次清理**（高频率）
- **时间**：12:30（午间 10 分钟）+ 21:00（晚间 10 分钟）
- **频率**：每天 2 次
- **优势**：
  - 收件箱永远不会堆积
  - 每次处理量少，心理负担小
  - 信息新鲜，决策更快
- **适合人群**：信息输入量大，追求零积压

**建立习惯的方法**：
1. **日历提醒**：在日历中创建重复事件
2. **环境触发**：设定"每天晚上回家后"或"午饭后"
3. **追踪打卡**：在周记中记录每日清理完成情况
4. **可视化激励**：统计连续清空天数

### 步骤 2：使用快速决策框架处理每条内容

清空收件箱的核心是**快速决策**，而非完美分类。目标是**每条内容 10-30 秒内完成决策**。

**快速决策三问**：

```
┌─────────────────────────────────────┐
│    收件箱中的一条内容               │
└──────────────┬──────────────────────┘
               ↓
        【问题 1】
    这需要采取行动吗？
               │
       ┌───────┴───────┐
       │               │
      Yes             No
       ↓               ↓
   【问题 2】     【问题 2-A】
 能在2分钟内    这是有价值的
 完成吗？       信息吗？
       │               │
   ┌───┴───┐       ┌───┴───┐
  Yes     No      Yes     No
   ↓       ↓       ↓       ↓
立即执行  归档    保存    删除
         到正确  到正确
         位置    位置
```

**决策框架详解**：

**分支 1：需要行动 + 2 分钟内完成 → 立即执行**
- **示例**：
  - "回复某人的消息" → 立即发送，记录到日记
  - "查询某个信息" → 立即搜索，结果保存或记录
  - "添加日历事件" → 立即添加，删除笔记
- **原则**：2 分钟法则（GTD 核心），避免小任务积压
- **处理后**：直接删除收件箱中的该条内容

**分支 2：需要行动 + 超过 2 分钟 → 归档为任务/项目**
- **决策子树**：使用 [[PARA分类决策树]]
  - 有截止日期 + 多步骤 → `02-项目/`
  - 有截止日期 + 单任务 → 添加到相关项目的任务列表
  - 无截止日期 + 持续关注 → `03-领域/` 的行动计划
- **示例**：
  - "学习 Docker" → 创建项目 `02-项目/学习项目/学习Docker/`
  - "优化晨间流程" → 添加到 `03-领域/健康管理/` 的改进计划
  - "写博客：时间管理方法" → 创建项目 `02-项目/个人项目/博客-时间管理/`

**分支 3：不需要行动 + 有价值 → 保存为参考/笔记**
- **决策子树**：判断内容性质
  - 外部资料（文章、教程）→ `04-资源/参考资料/`
  - 阅读摘要或书籍笔记 → `04-资源/阅读笔记/`
  - 个人洞察和思考 → 提炼成卡片 `10-卡片盒/`
  - 临时记录（会议、对话）→ `20-日志/日记/` 或项目文件夹
- **示例**：
  - "一篇关于 React 性能优化的文章" → `04-资源/参考资料/`
  - "读书时的一段感悟" → 提炼成 `10-卡片盒/观点卡片/`
  - "会议中讨论的三个要点" → `02-项目/XX项目/会议记录-20260122.md`

**分支 4：不需要行动 + 无价值 → 删除**
- **识别标准**：
  - 已过时的信息（如过期的活动通知）
  - 已解决的临时问题记录
  - 重复的内容（已在其他地方保存）
  - "可能有用"但无明确用途的内容
  - 模糊的想法，现在看来没有价值
- **原则**：**删除不等于浪费，保留无用信息才是浪费。**
- **心理障碍**：不要因为"我花时间记录了"而不舍得删除（沉没成本谬误）

**快速决策技巧**：

**技巧 1：10 秒计时器**
- 为每条内容设定 10 秒决策时间
- 超过 10 秒还不能决定 → 暂时跳过，放到最后处理
- 培养快速判断力，避免纠结

**技巧 2：默认删除原则**
- 当犹豫"要不要保留"时，默认选择删除
- 真正重要的信息会再次出现
- 信息可以重新找到，时间无法重来

**技巧 3：批量相似决策**
- 先浏览所有内容，识别相似类型
- 对相似内容应用相同规则
- 示例：5 篇技术文章 → 统一保存到 `04-资源/参考资料/技术/`

### 步骤 3：按内容类型批量处理

将收件箱内容按类型分组，批量处理提高效率：

**内容类型 1：快速笔记（灵感、想法、随手记）**

**特征**：
- 位于 `01-收件箱/快速笔记/`
- 通常是零散的句子或段落
- 记录时没有明确分类

**处理流程**：
```
浏览所有快速笔记
  ↓
识别类型：任务 / 想法 / 参考
  ↓
任务类 → 添加到项目任务列表或创建新项目
想法类 → 提炼成卡片（概念/方法/观点）或删除
参考类 → 补充到现有笔记或保存为资源
  ↓
删除收件箱中的原始快速笔记
```

**批量处理示例**：

假设收件箱中有 8 条快速笔记：

1. "Docker 容器网络配置问题" → 任务类
   - **决策**：添加到项目 `学习Docker` 的待办任务
   - **行动**：在项目笔记中添加 `- [ ] 研究容器网络配置`
   - **删除**：收件箱中的原始笔记

2. "原子化笔记的本质是降低认知负荷" → 想法类
   - **决策**：有价值的洞察，提炼成卡片
   - **行动**：
     - 打开 `10-卡片盒/概念卡片/原子化笔记.md`
     - 在"理论基础"部分添加这个洞察
     - 或创建新卡片 `10-卡片盒/观点卡片/原子化与认知负荷.md`
   - **删除**：收件箱中的原始笔记

3. "明天下午 3 点开会讨论 Q1 规划" → 任务类（2 分钟）
   - **决策**：立即行动
   - **行动**：添加到日历，在今日日记中记录
   - **删除**：立即删除

4. "可能有用的工具：Notion AI" → 参考类
   - **决策**：暂时无明确用途
   - **行动**：应用三问原则 → 无法回答"什么时候会用" → 删除
   - **删除**：直接删除

5-8. [其他笔记类似处理]

**内容类型 2：待处理资料（保存的文章、PDF、链接）**

**特征**：
- 位于 `01-收件箱/待处理/`
- 从外部保存的资料
- 使用 Web Clipper 或手动保存

**处理流程**：
```
浏览资料标题和来源
  ↓
应用资料三问原则：
  1. 能解决什么具体问题？
  2. 什么场景下会查看？
  3. 不保存能找回吗？
  ↓
有明确用途 → 归档到 04-资源/ 或项目文件夹
无明确用途 → 删除（信息随时可重新搜索）
高价值资料 → 立即阅读并提炼卡片
  ↓
删除收件箱中的资料或标记为已处理
```

**批量处理策略**：

**策略 A：按主题分组**
- 扫描所有待处理资料
- 识别主题集群（如 5 篇都关于"时间管理"）
- 同一主题资料批量归档到同一位置
- 如果某主题资料 ≥ 3 篇，考虑创建专题 MOC

**策略 B：按优先级排序**
- 紧急需要的资料（项目相关）→ 优先处理，归档到项目文件夹
- 学习计划相关 → 归档到对应领域或学习项目
- 一般性资料 → 快速评估，保留或删除

**示例**：

收件箱中有 12 篇待处理文章：
- 5 篇关于"Docker 技术" → 归档到 `02-项目/学习项目/学习Docker/resources/`
- 3 篇关于"时间管理" → 归档到 `04-资源/参考资料/效能提升/`
- 2 篇关于"React 性能" → 归档到 `04-资源/参考资料/技术/前端/`
- 2 篇无法明确分类或价值不大 → 删除

**内容类型 3：灵感闪念（创意点子、潜在写作主题）**

**特征**：
- 位于 `01-收件箱/灵感闪念/`
- 通常是一句话或简短描述
- 可能模糊、不完整

**处理流程**：
```
逐条阅读灵感内容
  ↓
判断是否仍然有价值（灵感会衰减）
  ↓
有价值 → 转化为项目、卡片或扩展笔记
  • 可执行的创意 → 创建项目
  • 有深度的洞察 → 提炼成观点卡片
  • 需要研究的主题 → 创建学习项目或 MOC
无价值 → 直接删除（灵感过期很正常）
  ↓
删除收件箱中的原始灵感笔记
```

**灵感处理示例**：

1. "写一篇关于 Zettelkasten 和 GTD 结合使用的文章"
   - **评估**：具体可行的写作计划
   - **行动**：创建项目 `02-项目/个人项目/博客-Zettelkasten与GTD/`
   - **删除**：收件箱中的灵感笔记

2. "时间块 + 能量管理 = 更高效的工作？"
   - **评估**：有潜力的观点，但需要验证
   - **行动**：创建卡片 `10-卡片盒/观点卡片/时间-能量矩阵假设.md`
   - **后续**：实践 2 周后补充案例
   - **删除**：收件箱中的灵感笔记

3. "做一个笔记软件对比评测"
   - **评估**：想法很好，但当前无时间和动力
   - **行动**：判断为"可能有用但不紧急" → 删除
   - **原则**：如果 3 个月后仍想做，灵感会再次出现

### 步骤 4：执行立即行动（2 分钟法则）

GTD 的核心原则：**如果一个任务可以在 2 分钟内完成，立即执行，不要归档为待办。**

**适合立即执行的内容**：
- 回复消息或邮件
- 添加日历事件
- 快速查询信息并记录
- 更新项目进度
- 标记任务为完成
- 简单的文件整理
- 发送提醒给他人

**立即行动的优势**：
- 避免小任务积压成山
- 减少大脑记忆负担
- 产生"完成感"正向激励
- 防止小任务变成"待办黑洞"

**实施方法**：

在清理收件箱时，设定"行动模式"：
1. 浏览收件箱，识别所有可 2 分钟完成的任务
2. 集中执行这些任务（5-10 分钟）
3. 任务完成后，直接删除收件箱中的对应笔记

**示例**：

**快速笔记**："回复 Alex 关于项目进度的问题"
- **决策**：2 分钟内可完成
- **行动**：打开消息应用，发送回复："进度 70%，预计周五完成"
- **记录**：在今日日记中添加"已回复 Alex 项目进度"
- **删除**：收件箱笔记

**快速笔记**："检查 Docker 官方文档关于网络模式的说明"
- **决策**：查询可在 2 分钟完成
- **行动**：
  - 打开 Docker 文档搜索"network modes"
  - 快速浏览，找到答案
  - 在项目笔记中记录要点
- **删除**：收件箱笔记

### 步骤 5：批量处理同类内容

将收件箱中相似的内容分组处理，提高效率：

**批量处理的优势**：
- 减少认知切换成本
- 应用相同决策规则
- 操作更流畅（如批量移动文件）
- 更容易发现主题模式

**批量处理策略**：

**策略 1：先分类，再处理**
```
第一遍：快速浏览，心理分类
  - 标记任务类 📋
  - 标记资料类 📄
  - 标记想法类 💡
  ↓
第二遍：按类型批量处理
  - 先处理所有任务类
  - 再处理所有资料类
  - 最后处理所有想法类
```

**策略 2：按处理时间分组**
```
2 分钟任务 → 立即集中执行（5-10 分钟）
  ↓
5-10 分钟任务 → 归档为项目任务，稍后执行
  ↓
长期任务/资料 → 归档到正确位置
```

**策略 3：按目标分组**
- 同一项目的多条内容 → 批量移动到项目文件夹
- 同一主题的多篇资料 → 批量添加相同标签，统一归档

**批量处理实例**：

**场景**：收件箱中有 15 条内容

**第一遍浏览**（2 分钟）：
- 5 条任务类 📋
- 6 条资料类 📄
- 3 条想法类 💡
- 1 条无价值 ❌

**第二遍处理**（13 分钟）：

**任务类批量处理**（5 分钟）：
1. "回复 John 邮件" → 立即执行（2 分钟）
2. "添加下周二会议到日历" → 立即执行（1 分钟）
3. "更新项目 A 进度" → 打开项目笔记更新（2 分钟）
4. "研究 Kubernetes 部署方案" → 添加到项目任务列表
5. "优化周回顾流程" → 添加到领域改进计划

**资料类批量处理**（5 分钟）：
- 3 篇 Docker 文章 → 批量移动到 `02-项目/学习Docker/resources/`
- 2 篇时间管理文章 → 移动到 `04-资源/参考资料/效能提升/`
- 1 篇 React 教程 → 移动到 `04-资源/参考资料/技术/前端/`

**想法类批量处理**（3 分钟）：
- "知识管理的本质是降低检索成本" → 提炼成观点卡片
- "卡片盒 + MOC = 个人维基百科" → 补充到现有笔记
- "做一个知识管理工具评测" → 评估后删除（暂无动力）

**无价值内容**（1 秒）：
- "测试笔记 123" → 删除

**结果**：15 条内容 → 13 分钟全部处理完毕 → 收件箱归零

### 步骤 6：为归档内容完善元数据

内容从收件箱移出后，需要补充完整的元数据，方便未来检索：

**最低元数据要求**：
```yaml
---
title: "笔记标题"
created: YYYY-MM-DD
tags: [标签1, 标签2]
---
```

**项目笔记元数据**：
```yaml
---
title: "项目名称"
status: "🚧 进行中"
priority: "🟡 中"
start_date: 2026-01-22
due_date: 2026-03-22
tags: [项目, 学习]
---
```

**资源笔记元数据**：
```yaml
---
title: "文章标题"
source: "来源网站"
url: "原文链接"
saved_date: 2026-01-22
tags: [参考, 技术, Docker]
---
```

**卡片盒笔记元数据**：
```yaml
---
title: "概念名称"
type: 概念卡片
created: 2026-01-22
tags: [卡片盒, 概念, 知识管理]
---
```

**快速补充方法**：
- 使用 Templater 插件自动填充日期
- 使用 QuickAdd 快速添加标签
- 批量添加标签（Tag Wrangler 插件）

### 步骤 7：验证收件箱为空并记录

**最终验证**：
```
打开 01-收件箱/ 文件夹
  ↓
检查三个子文件夹：
  - 快速笔记/ - 应该为空（除 README.md）
  - 待处理/ - 应该为空（除 README.md）
  - 灵感闪念/ - 应该为空（除 README.md）
  ↓
如果有遗漏 → 快速处理
  ↓
收件箱归零 ✅
```

**使用 Dataview 验证**：

在 `🏠 HOME.md` 或今日日记中添加查询：
```dataview
LIST
FROM "01-收件箱"
WHERE file.name != "README"
```

## 常见问题

### Q1：收件箱总是清不完，每天都有新内容怎么办？

A：这是捕获标准的问题，而非清理方法的问题

**根本原因**：
- 捕获了太多低价值信息
- 没有在捕获时进行初步筛选
- "先记下来再说"的习惯导致过度捕获

**解决方案**：

**1. 提高捕获门槛**

引入**捕获三问**（在记录前问自己）：
- **Q1**：这个信息能解决我当前的具体问题吗？
- **Q2**：我能明确描述将来使用它的场景吗？
- **Q3**：如果不记录，我会真的需要再次回忆它吗？

只有**至少 2 个"Yes"**才记录到收件箱。

**2. 分级捕获策略**

**A 级信息**（必须捕获）：
- 与进行中项目直接相关
- 独特的洞察和灵感
- 重要的任务和约定
- 难以二次找到的信息

**B 级信息**（选择性捕获）：
- 可能有用的资料
- 模糊的想法
- 一般性任务

**C 级信息**（不捕获）：
- 可以轻易搜索到的通用知识
- "可能有用"但无明确场景
- 重复的信息

**调整前后对比**：

| 指标 | 调整前 | 调整后 |
|------|--------|--------|
| 每日捕获量 | 15 条 | 5 条 |
| 每日清理时间 | 25 分钟 | 8 分钟 |
| 内容保留率 | 40% | 80% |
| 后续利用率 | 20% | 60% |

**结论**：捕获少但精，清理更高效，利用率更高。

### Q2：清理收件箱时总是犹豫不决，如何提高决策速度？

A：使用决策辅助工具和明确的决策规则

**问题分析**：
- 追求完美分类，纠结"这应该放哪里"
- 不确定内容价值，担心"删了以后需要"
- 缺少明确的决策标准

**解决方案**：

**工具 1：打印决策清单**

将以下清单打印或保存为快捷笔记，清理时参考：

```markdown
📋 收件箱清理决策清单

【步骤 1】这需要行动吗？
  → No → 【步骤 2】
  → Yes → 2分钟内完成？
     → Yes → 立即执行
     → No → 归档到项目/领域

【步骤 2】这是外部资料吗？
  → Yes → 04-资源/
  → No → 【步骤 3】

【步骤 3】这是独立的洞察/方法？
  → Yes → 10-卡片盒/
  → No → 【步骤 4】

【步骤 4】与现有笔记相关？
  → Yes → 补充到现有笔记
  → No → 删除

⏱️ 每条内容决策时间 < 30 秒
🚫 超过 30 秒 → 跳过，最后处理
```

**工具 2：使用计时器强制决策**

- 为每条内容设置 20 秒倒计时
- 时间到必须做出决策
- 培养快速判断的直觉

**工具 3：默认行为规则**

当无法决策时，使用默认规则：
- **默认规则 1**：犹豫是否删除 → 默认删除
- **默认规则 2**：犹豫归档位置 → 默认选择 `04-资源/参考资料/`
- **默认规则 3**：犹豫是否提炼 → 默认先保存，周回顾时再提炼

**工具 4：延迟决策队列**

- 遇到难以快速决策的内容 → 打上标签 `#延迟决策`
- 继续处理其他内容
- 最后集中处理 `#延迟决策` 内容
- 如果仍然无法决策 → 删除（说明不够重要）

**实践结果**：
- 决策时间从平均 45 秒降到 15 秒
- 清理总时间从 25 分钟降到 10 分钟
- 决策焦虑大幅减少

### Q3：2 分钟法则如何把握？有些任务看起来很快但实际很慢。

A：明确 2 分钟法则的适用范围和判断标准

**2 分钟法则的精确定义**：
- **计时起点**：从当前状态到任务完全完成的时间
- **包含时间**：启动工具、执行操作、验证结果的总时间
- **不包含**：思考、计划、准备工作

**适合 2 分钟法则的任务**：

✅ **通信类**（最常见）：
- 回复短消息或邮件（< 3 句话）
- 发送提醒或通知
- 转发资料给他人

✅ **记录类**：
- 添加日历事件
- 记录简单想法到日记
- 更新项目进度状态

✅ **查询类**：
- 搜索简单信息（如会议时间）
- 查阅文档获取答案
- 检查任务完成状态

✅ **简单整理类**：
- 文件重命名
- 添加标签
- 移动单个文件

**不适合 2 分钟法则的任务**：

❌ **需要思考的任务**：
- "写回复邮件"（需要组织语言）→ 归档为任务
- "设计 XX 方案"（需要深度思考）→ 创建项目

❌ **需要工具的任务**：
- "剪辑视频片段"（需要打开视频软件）→ 归档
- "修改代码 Bug"（需要启动 IDE）→ 归档

❌ **多步骤任务**：
- "整理上周的照片"（需要多次操作）→ 归档
- "清理电脑桌面"（需要逐个文件处理）→ 归档

**判断技巧**：

**技巧 1：想象操作流程**
- 在脑中快速过一遍操作步骤
- 步骤 > 3 个 → 不适合立即执行

**技巧 2：检查依赖**
- 需要打开额外工具/软件 → 不适合
- 需要其他人配合 → 不适合
- 纯粹在当前环境完成 → 适合

**技巧 3：设置 2 分钟计时器实测**
- 对不确定的任务实际计时
- 多次实测后建立时间直觉
- 记录"任务-实际耗时"对照表

**边界案例处理**：

**案例 A**："回复客户邮件"
- **初判**：似乎可以 2 分钟完成
- **实测**：需要查阅项目资料、组织专业表达 → 实际需要 10 分钟
- **结论**：不适合立即执行
- **行动**：添加到今日待办 `- [ ] 回复客户邮件（预计 15 分钟）`

**案例 B**："添加周五的会议到日历"
- **初判**：肯定可以 2 分钟完成
- **实测**：打开日历、创建事件、设置提醒 → 实际 1 分钟
- **结论**：适合立即执行
- **行动**：立即打开日历添加，删除收件箱笔记

### Q4：有些内容不知道是"任务"还是"想法"，如何区分？

A：使用明确的判断标准

**任务的特征**：
- ✅ 包含动词（做、写、学、整理、联系）
- ✅ 有可观察的完成状态
- ✅ 需要消耗时间和精力去执行
- ✅ 完成后可以打勾 ✓

**想法的特征**：
- ✅ 包含概念、观点、洞察
- ✅ 是对世界的理解或看法
- ✅ 不需要"执行"，需要"记录和连接"
- ✅ 可能产生新的任务或项目

**判断标准**：

**问自己**："这是一个 **to-do**（要做的事）还是 **to-know**（要记住的知识）？"

**对比示例**：

| 收件箱内容 | 类型 | 判断依据 | 处理方式 |
|-----------|------|---------|----------|
| "学习 Docker 网络配置" | 任务 | 包含动作"学习"，需要执行 | 创建学习项目 |
| "Docker 单一职责原则" | 想法/概念 | 是一个原则/概念 | 创建概念卡片 |
| "写一篇关于时间管理的博客" | 任务 | 包含动作"写"，需要执行 | 创建写作项目 |
| "时间管理的本质是能量管理" | 想法/观点 | 是一个洞察/观点 | 创建观点卡片 |
| "整理上周的会议记录" | 任务 | 包含动作"整理"，需要执行 | 添加到今日待办 |
| "会议中讨论的三个关键决策" | 想法/记录 | 是信息记录，不需要执行 | 归档到项目文件夹 |

**快速判断法**：

**方法 1：加上"我要"测试**
- 前面加上"我要" → 通顺 → 任务
  - "我要学习 Docker" ✅ → 任务
  - "我要时间管理的本质是能量管理" ❌ → 不通顺 → 想法

**方法 2：想象完成状态**
- 能明确想象"完成"是什么样子 → 任务
- 无法想象"完成"状态 → 想法

**方法 3：时间轴测试**
- 可以放在日程表或任务列表中 → 任务
- 不适合放在时间轴上 → 想法

**混合情况处理**：

有些内容既包含任务又包含想法：
```markdown
"学习费曼学习法，并用它来学习 Python"
```

**拆解处理**：
- **任务部分**："学习费曼学习法" + "用它学习 Python"
  - 创建学习项目或添加到现有项目任务
- **想法部分**："费曼学习法" 这个概念
  - 如果不了解，添加到学习清单
  - 如果已了解，不需要额外行动

### Q5：如何避免收件箱变成"第二个待办清单"？

A：明确收件箱和待办清单的边界

**核心区别**：

| 特性 | 收件箱 | 待办清单 |
|------|--------|----------|
| **定位** | 临时中转站 | 执行清单 |
| **内容** | 未分类的原始捕获 | 明确的行动任务 |
| **停留时间** | < 24 小时 | 直到完成 |
| **处理方式** | 分类和归档 | 执行和打钩 |
| **数量** | 必须清空（目标 0） | 可以积累（控制在合理范围） |

**错误模式**：

❌ **反模式 1：在收件箱中标记任务优先级**
```markdown
01-收件箱/快速笔记/
  ├─ 任务A-高优先级.md
  ├─ 任务B-中优先级.md
  └─ 任务C-低优先级.md
```
**问题**：收件箱变成了任务管理工具
**正确做法**：将任务归档到项目的任务列表，在项目笔记中标记优先级

❌ **反模式 2：在收件箱中创建子文件夹分类**
```markdown
01-收件箱/
  ├─ 待处理/
  │   ├─ 工作相关/
  │   ├─ 个人相关/
  │   └─ 学习相关/
```
**问题**：收件箱变成了分类系统
**正确做法**：快速笔记直接放在 `快速笔记/` 下，清理时归档到正确位置

❌ **反模式 3：在收件箱笔记中添加详细元数据**
```yaml
---
title: "学习 Docker"
priority: 高
status: 待开始
due_date: 2026-02-01
tags: [学习, 技术, 待办]
---
```
**问题**：在收件箱中编辑和完善笔记
**正确做法**：快速捕获，元数据在归档时再补充

**正确模式**：

✅ **收件箱的正确用法**
```markdown
01-收件箱/快速笔记/
  ├─ 2026-01-22-1430.md （"学习 Docker 网络"）
  ├─ 2026-01-22-1528.md （"时间块+能量管理"）
  └─ 2026-01-22-1645.md （"回复 John 邮件"）
```

**每天晚上清理**：
1. 第 1 条 → 创建项目 `02-项目/学习项目/学习Docker/`，删除收件箱笔记
2. 第 2 条 → 提炼为观点卡片，删除收件箱笔记
3. 第 3 条 → 立即执行，删除收件箱笔记

**第二天早上**：
- 收件箱为空
- 项目笔记中有明确的任务列表
- 卡片盒中有新的观点卡片

**执行原则**：

**原则 1：收件箱只有 3 个标准文件夹**
- `快速笔记/` - 随手记录
- `待处理/` - 保存的资料
- `灵感闪念/` - 创意点子

不创建额外子文件夹，不进行二次分类。

**原则 2：收件箱中的笔记不添加复杂元数据**
- 捕获时：最多添加一个标签（如 `#待整理`）
- 归档时：在目标位置补充完整元数据

**原则 3：收件箱每日必须清空**
- 设定硬性规则：每天结束前收件箱归零
- 如果时间不够，快速决策：保留 or 删除
- 不允许内容在收件箱中过夜（除非特殊情况）

**原则 4：待办任务立即转移**
- 识别为任务的内容，立即转移到：
  - 项目任务列表
  - 今日日记的待办区
  - 领域的行动计划
- 不在收件箱中标记"待办"

**监控机制**：

每周回顾时检查：
```dataview
TABLE
  file.ctime as "创建时间",
  date(today) - file.ctime as "停留天数"
FROM "01-收件箱"
WHERE file.name != "README"
SORT file.ctime ASC
```

- 如果有内容停留 > 3 天 → 系统异常，需要调整习惯
- 如果收件箱长期为空 → 系统健康 ✅

```dataview
TABLE file.ctime as "捕获时间", file.folder as "位置"
```

**查询 2：超期未处理内容**
```dataview
TABLE
  file.ctime as "捕获时间",

FROM "01-收件箱"

SORT file.ctime ASC
```

```dataview
TABLE
  收件箱清理 as "完成",

WHERE date(file.name) >= date(today) - dur(7 days)
  AND 收件箱清理
SORT file.name DESC
```

```dataview
TABLE
  length(rows) as "数量"
FROM "01-收件箱"
WHERE file.name != "README"
GROUP BY date(file.ctime).week
SORT key DESC
LIMIT 4
```
